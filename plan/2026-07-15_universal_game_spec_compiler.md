# 通用游戏规格编译器重构开发计划

状态：复核后冻结，A01 已完成，下一项 A02
基线：`v0.1.0` / `1a74408680cc73f07440228068139218546fc458`
执行入口：`plan/dev/README.md`

## 1. 目标

将现有“固定设计选项 + 大量弱类型 JSON + 分段式流水线”演进为通用、可验证、可迁移的游戏规格编译系统。系统需要覆盖不同结构的游戏，而不是围绕某一个验证项目建立隐含模型。

核心结果：

1. 一个引擎无关、类型化、可版本迁移的 `GameSpec` 规范，作为设计到开发的唯一权威状态。
2. 一个按能力组合的决策图。类型或题材标签只能帮助检索，不能决定核心结构。
3. 一个单写者、带修订号和内容哈希的状态演进模型。所有人工和 AI 修改都通过可审计补丁进入。
4. 一组确定性编译器和质量门禁，将同一份 `GameSpec` 编译为步骤 00–14 的阶段产物。
5. 一套跨结构验证矩阵。植物大战僵尸类项目只是“通道防守”样例之一，不拥有核心字段特权。

## 2. 不做的事情

1. 不更改产品面向用户的功能范围、项目内容设计或现有步骤编号。
2. 不重新引入 CrewAI、多角色互审、AI 投票或 AI 自主选择流程。
3. 不允许 AI 直接写权威状态、判定自己通过、跳过验证或静默降级。
4. 不在第一阶段一次性替换现有 `ProjectState`、保存格式和流水线。
5. 不把 Unity、塔防、卡牌、叙事等引擎或类型专属概念放入通用核心。

## 3. 当前问题审计

### 3.1 设计状态不是稳定的领域模型

`ProjectState` 以节点、清单和 `serde_json::Value` 承载大量语义。它适合现有 UI 编辑，但无法稳定表达动作前置条件、状态迁移、资源守恒、场景验收和跨步骤引用。

### 3.2 类型规则已渗入通用流水线

`semantic_pipeline.rs` 直接包含塔防和叙事解谜数据，类型识别结果继续驱动需求节点和任务。这会让验证项目反向塑造核心，且无法证明对结构不同的项目有效。

### 3.3 交接层承担过多职责

`handoff.rs` 同时负责领域投影、质量统计、多格式渲染、文件输出、跨层规则和兼容读取。弱类型值在这些转换中反复被推断，错误通常到后续步骤才暴露。

### 3.4 AI 补全没有成为受控的真实数据流

现有 AI 访谈包含 route、mapping、summary 等角色化模块，但缺少统一的“候选补丁 -> 确定性验证 -> 用户确认 -> 单写者提交”闭环。即使调用了模型，也难证明结果进入了后续权威产物。

### 3.5 00–14 的外壳可保留，内部契约需要重建

分段执行、检查点、恢复和阶段独立展示是正确方向。问题不是步骤数量，而是步骤之间传递的语义不够强、质量门禁不够早。

## 4. 目标架构

### 4.1 GameSpec 核心

新增独立 crate `adm-new-game-spec`，只依赖基础序列化与通用工具，不依赖 UI、AI、流水线、Unity 或具体游戏类型。

依赖方向固定为“应用层 -> 设计/流水线 -> GameSpec -> 基础库”。`adm-new-game-spec` 不依赖 `adm-new-contracts`，避免领域模型与 IPC DTO 形成环；后续 DTO 只做显式转换或只读视图。

核心聚合包含：

- `SpecIdentity`：规范版本、项目 ID、修订号、父修订哈希。
- `ProjectIntent`：体验承诺、受众、平台、范围、约束和成功指标。
- `CapabilityProfile`：空间、时间、控制、参与者、信息、进度、内容、联网等正交能力轴。
- `EntitySpec` / `ComponentSpec` / `RelationshipSpec`：对象、属性和关系。
- `ActionSpec`：参与者、输入、前置条件、效果、成本、冷却和反馈。
- `StateMachineSpec`：状态、事件、守卫、迁移和失败状态。
- `ResourceSpec`：来源、消耗、容量、交换和守恒约束。
- `SpaceSpec` / `TimeSpec` / `InteractionSpec`：空间拓扑、时间模型与交互通道。
- `ContentSpec` / `PresentationSpec`：内容集合、视听意图和资产需求。
- `TechnicalConstraints`：目标引擎、平台、性能、存档和可访问性约束。
- `AcceptanceScenario`：可执行或可人工复核的 Given/When/Then 场景。
- `TraceLink`：意图、决策、规格、任务、测试、资产和产物之间的引用。
- `ExtensionBlock`：带命名空间和版本的能力扩展，承载类型专属字段。

所有实体 ID 使用受验证的新类型；核心集合使用 `BTreeMap` 或在规范化时按 ID 排序。步骤、对话分支、效果栈等具有顺序语义的序列必须保留原顺序，不能为了哈希稳定而排序。当前内容哈希不包含自身哈希字段，但包含 schema 版本、修订号、父修订哈希和全部语义内容。

核心晋升规则：一个字段进入核心前，必须被至少三个结构明显不同的样例共同需要；否则保留在能力扩展中。

### 4.2 能力组合，不以类型分支

决策图由能力谓词激活，例如：

- `spatial.topology = lane | grid | continuous | graph | none`
- `time.progression = realtime | turn_based | event_driven | hybrid`
- `control.cardinality = single | party | squad | indirect`
- `content.generation = authored | procedural | mixed`
- `participants.mode = solo | local_multi | networked`

“塔防”“三消”“视觉小说”等仅作为搜索标签或样例名称。编译器只读取能力和规格，不读取类型名称来选择硬编码实现。

### 4.3 单写者规格仓库

所有变更统一为 `SpecPatch`：

- 指定 `base_revision` 和 `base_hash`，拒绝陈旧写入。
- 每个操作声明路径、旧值预期、新值、来源、理由和证据。
- 先在内存副本应用，再执行 schema、引用、语义和影响验证。
- 只有一个 `SpecStore` 可以原子提交，提交后生成新修订和不可变审计记录。
- 人工编辑、模板、迁移器和 AI 使用同一入口，不存在旁路写入。

规格快照属于项目保存作用域，只保存相对资源引用；机器路径和 AI 凭证继续由本机绑定层管理，不能进入可迁移规格或审计日志。

### 4.4 AI 的边界

AI 只产生 `CandidateSpecPatch`，不能直接产生“已通过”状态。

真实闭环：

1. `PromptPack` 只提供相关子图、未决问题、允许路径和输出 schema。
2. 模型返回候选补丁、置信度、依据和未解决项。
3. Rust 端拒绝越界路径、未知引用、删除保护字段和 schema 不匹配。
4. 确定性门禁计算影响面、冲突和覆盖率。
5. 低风险补丁由用户批量确认；高风险补丁逐项确认。
6. 单写者提交并记录非敏感模型配置 ID、输入/输出摘要哈希和验证证据；不记录密钥、认证头或未经脱敏的原始提示。

禁止 AI 投票、角色互相说服、AI 选择是否执行某步骤，以及用第二次 AI 调用替代确定性验证。

### 4.5 编译器和投影

`GameSpec` 是权威输入，现有 UI 节点、交接 Markdown、开发任务和流水线产物都是可重建投影。投影必须是纯函数或显式 IO 编译器；同一输入和版本必须产生相同语义结果。

建议边界：

- `adm-new-game-spec`：领域模型、补丁、验证、版本迁移。
- `adm-new-design`：决策图、编辑命令、旧状态兼容投影。
- `adm-new-pipeline`：步骤编译器、检查点和产物编排。
- `adm-new-ai`：候选补丁生成，不拥有状态。
- `adm-new-storage`：规格快照、审计日志和原子提交。
- `adm-new-contracts`：跨进程 DTO，不复制领域行为。

## 5. 设计流程重构

现有 D1–D4 外部入口保留，内部重解释：

| 阶段 | 新职责 | 必须输出 |
|---|---|---|
| D1 | 项目意图与硬约束 | `ProjectIntent`、开放问题、不可违反约束 |
| D2 | 能力画像与动态决策图 | `CapabilityProfile`、激活模块、决策依赖 |
| D3 | 玩法和内容规格合成 | 动作、状态、实体、资源、空间、时间和反馈契约 |
| D4 | 场景验证与规格冻结 | 验收场景、追踪矩阵、冻结哈希和未决风险 |

原 16 个领域、103 个节点、515 个清单项降级为知识库和编辑模板。只有命中能力谓词且对当前规格有未满足约束的节点才激活；未激活内容不会拉低完成度。

## 6. 流水线 00–14 重解释

| 步骤 | 新职责 | 确定性门禁 |
|---|---|---|
| 00 | 输入快照与基线 | 身份、版本、完整输入、无隐式父目录依赖 |
| 01 | 意图与约束编译 | 体验承诺、范围、平台约束无矛盾 |
| 02 | 能力画像与决策图 | 激活理由完整、无类型名硬分支 |
| 03 | 动作与交互契约 | 每个核心循环可追踪到动作、条件、效果和反馈 |
| 04 | 世界、实体、状态和资源 | 引用有效、迁移闭合、资源约束可解释 |
| 05 | 验收场景与测试模型 | 每项关键承诺至少一个正向和失败场景 |
| 06 | GameSpec 冻结与交接 | 无阻塞问题、追踪闭合、冻结哈希稳定 |
| 07 | 视觉方向与图像任务 | 提示词来源可追踪、尺寸/用途/风格约束完整、图像质量门禁通过 |
| 08 | 运行时架构编译 | 系统边界来自能力，不由类型标签决定 |
| 09 | 资产与内容生产计划 | 每项资产有用途、规格、预算、依赖和验收 |
| 10 | 实现任务图 | 输入输出、所有权、依赖、回滚和验证齐全 |
| 11 | 程序生成与集成 | 只写声明范围，编译/测试/静态检查通过 |
| 12 | 资产集成 | 元数据、引用、许可、格式和运行时加载通过 |
| 13 | 场景验证与修正 | 验收场景、性能、可访问性和回归证据闭合 |
| 14 | 打包与发布证据 | 可重现构建、完整性、启动 smoke 和发布清单通过 |

UI 继续只展示当前步骤产物；图像等富媒体不得泄漏到其他步骤视图。

## 7. 质量体系

不再使用“AI 自评分”作为发布结论。每一步的 100 分评分由可复算指标组成：

- 契约完整度 25 分。
- 引用和追踪闭合度 20 分。
- 矛盾与不变量 20 分。
- 验收场景覆盖 20 分。
- 项目定制和证据质量 15 分。

存在阻塞错误时总分封顶 69；存在静默 fallback、未知引用、未确认高风险 AI 写入或伪造证据时直接失败。主观体验评价可以作为附加评审，不覆盖确定性门禁。

## 8. 跨结构验证矩阵

最终至少包含以下样例：

1. 通道防守：验证持续生成、路径、放置、资源和波次。
2. 三消：验证网格、局部规则、组合效果和关卡目标。
3. 分支叙事：验证图结构、条件、变量和多结局。
4. 回合战术：验证单位、行动点、网格、顺序和胜负条件。
5. 动作 Roguelite：验证实时输入、碰撞、程序内容和局外成长。
6. 卡组构筑：验证卡牌数据、牌区、效果栈和随机性。
7. 经营建造：验证时间、生产链、容量和长期进度。
8. 网络协作：验证权威性、同步、断线和重连约束。

植物大战僵尸项目只映射到第 1 个样例，用于覆盖验证，不作为字段命名或默认模板来源。

## 9. 迁移与兼容策略

1. 使用 `game_spec_v2` 功能开关，默认关闭，先双轨生成不接管 UI。开关具有项目级持久状态；环境变量只允许用于开发测试，不能决定便携项目的长期格式。
2. 从现有 `ProjectState` 生成只读 `GameSpec` 候选和差异报告，不反向覆盖。
3. 达到跨样例门禁后，允许新建项目默认写 V2；旧项目继续兼容读取。
4. 一段稳定期内同时生成旧交接和 V2 投影，比较语义差异。
5. 只有保存迁移、回滚、便携包和真实项目验证全部通过后，才移除旧写路径。

## 10. 执行阶段和停止门

| 阶段 | 原子任务 | 停止门 |
|---|---|---|
| 基础 | A01–A02 | 类型核心不含引擎/类型专属字段；四类样例可往返并通过验证 |
| 组合 | A03–A04 | 决策图确定性；陈旧补丁、越界写入和并发冲突全部拒绝 |
| AI | A05 | 模型无法绕过 schema、确认和单写者；调用证据可追踪 |
| 设计接入 | A06 | D1–D4 双轨输出稳定；旧项目无行为回归 |
| 流水线 | A07–A08 | 00–14 全链路只消费冻结规格或显式兼容投影 |
| 评估 | A09 | 八类样例和重复运行门禁通过；无样例专属核心泄漏 |
| 发布 | A10 | 迁移、回滚、独立性和便携构建证据完整 |

任何阶段不满足停止门，不进入下一阶段，也不通过扩大 AI 自由度来补偿模型缺陷。

## 11. 计划复核与优化记录

本计划写入后已按当前 crate 依赖、保存模型、独立性门禁和 00–14 执行器完成一轮复核，直接合入以下修正：

1. 将大爆炸替换改为 `game_spec_v2` 项目级开关、影子生成、语义差异报告和阶段性默认切换。
2. 固定新领域 crate 的依赖方向，避免把行为塞回 IPC DTO 或与现有设计 crate 形成循环依赖。
3. 区分无序 ID 集合与有序语义序列，防止“稳定哈希”破坏动作效果、状态迁移或叙事顺序。
4. 明确规格和日志禁止机器路径、凭证和原始敏感提示，继续满足跨电脑独立性要求。
5. 将跨结构样例提前到 A01，而不是系统完成后才发现核心已被第一个验证项目塑形。
6. 为每阶段增加停止门和删除式回滚边界，后续任务不能以未验证的占位实现冒充完成。

复核结论：原子任务顺序与总体阶段一致，无循环依赖；A01 已在不改变现有运行行为的前提下完成。

## 13. 完成定义

1. 新项目可从空状态完成 D1–D4 和 00–14，无隐式读取旧项目或父目录数据。
2. AI 补全在产物中有明确使用证据，并且关闭 AI 时流程仍可人工完成。
3. 八类样例均通过 schema、引用、不变量、场景、追踪和重复性门禁。
4. 通道防守样例表现优秀，但核心源码中不存在品牌名或类型专属默认分支。
5. 旧项目可读取、迁移、回滚；迁移失败不会损坏原保存。
6. 完整便携验证、EXE smoke 和跨电脑索引边界继续通过。
