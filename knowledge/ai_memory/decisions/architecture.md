## 2026-06-19：GUI 重构为主框架 + 标签切换

**决策**：将 `CommercialDesignApp(tk.Tk)` 改为 `CommercialDesignApp(tk.Frame)`，嵌入新的 `MainWindow(tk.Tk)` 主框架。顶部加标签栏切换"设计工作台" / "开发流水线"，不弹独立子窗口。

**理由**：
- 用户明确要求"打开时全屏"、"参考 VSCode 界面逻辑"
- 标签切换比独立 Toplevel 窗口更符合 IDE 风格
- 避免多窗口焦点管理问题
- 设计工作台和流水线共享底部面板（日志 + AI 对话），切换时上下文同步

**否决方案**：
- ❌ 独立 Toplevel 子窗口：会失去焦点，用户体验差，与 VSCode 风格不符
- ❌ 直接在 CommercialDesignApp 内加标签：代码耦合严重，难以维护

**影响范围**：
- 需修改 `core/ui/app_window.py`（CommercialDesignApp 改为 tk.Frame）
- 需新增 `core/ui/main_window.py`（主框架）
- 需修改 `core/ui/gui_app.py`（入口切换到 MainWindow）

## 2026-06-19：记忆系统放在 knowledge/ai_memory/

**决策**：AI 记忆系统存放在 `knowledge/ai_memory/` 目录，跟随 git 版本管理，不放在项目外的个人目录。

**理由**：
- 项目级知识应该共享，团队/多 AI 实例看到同样的记忆
- `knowledge/` 目录本身就是知识层，AI 记忆是知识的一种
- 遵守 CLAUDE.md 规则：不在根目录新建文件夹
- 与 Claude Code 的个人记忆系统（`~/.claude/projects/.../memory/`）互补而不重叠

**否决方案**：
- ❌ 放在根目录 `memory/`：违反 CLAUDE.md 规则
- ❌ 放在 `.claude/`：不跟随 git，无法共享
- ❌ 放在 `ucos/`：ucos 是游戏设计会话的 episodic 记忆，职责不同

**影响范围**：
- `knowledge/ai_memory/` 纳入 git 版本管理
- CLAUDE.md 顶部引用 `knowledge/ai_memory/INDEX.md`

## 2026-06-19：Unity 配置未完成时弹窗提示而非禁用按钮

**决策**：步骤03及以上运行时，若 Unity 路径未配置，点击"运行"后弹出 `messagebox.showwarning()`，精确提示缺失字段，不允许执行。而不是提前禁用"运行"按钮。

**理由**：
- 用户明确要求"不允许运行，弹出提示，哪个没配置就提示哪个"
- 弹窗能给出详细错误信息（development_path 缺失 vs editor_path 缺失）
- 禁用按钮会让用户不知道为什么不能点
- 引导用户点击"Unity配置"完成设置

**否决方案**：
- ❌ 直接禁用按钮：用户不知道原因
- ❌ 显示警告标记但允许继续：会导致步骤执行失败，浪费时间

## 2026-06-19：底部面板日志不随步骤切换清空

**决策**：日志面板显示本次会话的全部输出，不随用户切换步骤而清空。

**理由**：
- 用户可能需要对比多个步骤的日志
- 全局日志符合 IDE 风格（VSCode 的输出面板也是全局的）
- 如果需要清空，可以加"清空日志"按钮，由用户主动触发

**否决方案**：
- ❌ 切换步骤时清空：丢失上下文，对比日志困难

## 2026-07-10：跨窗口问题记录统一使用 plan/fixplan

**决策**：二次开发过程中产生的问题、根因、解决方案和验证结果，统一记录在 `plan/fixplan/`；当前窗口和其他窗口都使用同一目录。

**理由**：
- 让多个开发窗口共享同一份问题处理上下文
- 将修复记录与 Rust 二次开发计划集中管理，便于后续检索和复盘
- 独立 Markdown 文件降低多窗口同时追加造成的冲突

**记录约定**：
- 独立问题使用 `YYYY-MM-DD_NNN_<short-slug>.md` 文件
- 记录问题、复现条件、根因、解决方案、验证和后续事项
- 不记录 API 密钥、访问令牌或其他敏感配置

## 2026-07-15：通用能力组合与单写入器替代类型模板和多角色工作流

**决策约束**：NEWrust 后续设计重构必须以引擎无关的通用 Game IR 为核心，通过确定性能力规则组合项目所需模块。PvZ 只作为回归样例之一，不能决定核心 schema、默认值、路由、评分或步骤语义。

**AI 边界**：
- AI 只能生成结构化候选补丁、候选方案、代码或媒体产物，不能直接修改规范主状态。
- 规范存储与编译器是唯一写入者，负责 schema、引用、依赖、状态机、资源流、追踪和验收门禁。
- 禁止 CrewAI 式多角色对话、角色互审、AI 投票、AI 自主编排和以 AI 自评作为正式通过依据。
- AI 调用失败或证据不足时必须阻断或显式降级，不能用通用占位结果伪装成功。

**通用模型**：核心只表达项目意图、能力画像、实体/组件、动作、条件/效果、状态、资源、内容、交互、呈现、技术约束、场景验收和追踪关系。空间、战斗、经济、叙事、多人、程序化内容等通过可选模块激活；类型/题材只作标签或初始建议。

**验证原则**：建立多个结构差异显著的黄金样例、重复运行评测和变异测试。任何新核心概念必须证明可跨多个项目族复用，否则保留在能力扩展中。PvZ 只能验证组合覆盖，不能反向塑造通用内核。

**参考边界**：只借鉴经 2026-07-15 核验高于 5K stars 的 Spec Kit、OpenSpec、Bevy、GDevelop、Promptfoo、Argo Workflows、Dagster、CUE 等项目的规格、组合、DAG、血缘和评测思想，不直接引入其运行时或多代理模式。

**落地边界（2026-07-15-002）**：
- 新领域内核固定为 `adm-new-game-spec`，依赖方向为应用层到设计/流水线，再到 GameSpec 和基础库；GameSpec 不依赖 IPC DTO 或应用服务。
- 跨对象字段使用 `SpecRef` 明确目标类型，禁止裸 ID 依赖上下文猜测实体、资源或状态。
- 无序 ID 集合使用稳定键顺序；有序效果、迁移、分支和步骤保持原顺序，规范化不得改变语义。
- V2 先通过项目级开关进行影子生成和语义差异比较，不直接接管旧保存或现有 UI。
- A01 已完成且未接入运行路径；A02 完成确定性验证和哈希前，不允许进入状态写入和流水线迁移。
