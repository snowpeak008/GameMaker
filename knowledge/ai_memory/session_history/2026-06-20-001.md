## 做了什么

### 1. 修复设计项目保存流水线

执行对象保存（`core/engines/execution_objects/design_project.py`）连续报错，逐一修复：

| 报错 | 修复 |
|------|------|
| no save loaded | `app_window.save_project` 先调 `ensure_current_save()` |
| elevated_confirm 不匹配 | `type_registry.py`：design_project → `normal_confirm` |
| approve 失败 | evidence 补 `confirmed: True` |
| start_execution 要求 drift/conflict check | approve 后补 `run_pre_execution_drift_check` + `check_concurrency_conflicts` |
| `verify_execution` 不存在 | 改为 `store.verify()`，补6个 shared fact 字段 |
| `store._hash_content` 不存在 | 改用模块级 `stable_hash()` |
| `conflict_blocked` 无法取消 | `workflow.py` 新增 `force_cancel()` 方法，跳过终态强制置 cancelled |

### 2. 存档管理界面重构

- 新建 `core/ui/save_manager_dialog.py`：独立 `SaveManagerDialog(tk.Toplevel)` 类
- 功能：新建空白存档、保存到选中存档、加载、删除、打开目录、刷新
- `_save_has_design_project()` 过滤：只显示含 design_project 执行对象的存档槽，隐藏纯流水线存档
- `app_window.save_project()` 简化为3行，调 `SaveManagerDialog(self)`
- 清理了今天调试产生的垃圾存档 `save_20260620_133102_f01943`（6.5MB 垃圾执行对象）
- 清理了 `sandbox/workspace/saves/` 下6个旧流水线存档，工作区置空

### 3. 流水线项目引擎选择

**`core/runtime/preflight.py`**
- 新增 `SUPPORTED_ENGINES / ENGINE_LABELS / ENGINE_PATH_LABELS` 常量
- `load_project_settings()` 加 `project_engine` + `custom_engine_name`
- preflight 按引擎分支：Unity 检查 Assets/；Unreal/Godot 只验路径；custom 可选

**`core/ui/unity_config_dialog.py`**（重构为 `ProjectConfigDialog`）
- 顶部引擎 Combobox（Unity/Unreal/Godot/自定义）
- 布局固定顺序，不用 `after=` 动态插入，避免布局混乱

- `_dev_section` 一次性 pack，不在引擎切换时重复调用
- 所有引擎均显示编辑器路径；custom 额外显示说明提示
- 按钮移至右下角

**`core/ui/pipeline_panel.py`**
- `_render_detail()` 显示「当前引擎：xxx」

## 关键发现

1. **runtime_root 是 `sandbox/workspace`**，不是 PROJECT_ROOT
   - 存档槽在 `sandbox/workspace/saves/`，与流水线共享
   - 设计工作台的 save_manager 操作均以此为基准

2. **旧流水线存档污染设计存档列表**
   - `save_20260609_*` 等是流水线产生的，不含 design_project 数据
   - 修复：用 `_save_has_design_project()` 过滤

3. **执行对象保存失败根本原因**
   - 历次失败在 `approved` 等无公开取消路径的状态留下残留
   - 新增 `force_cancel()` 彻底解决
