# 2026-07-15-001

## 摘要

重新审查 NEWrust 当前设计方法，确认现有方案存在 PvZ/塔防测试样例反向塑造通用模型的问题，并将后续重构约束收敛为“通用能力组合、单一规范状态、确定性编译和验证、AI 仅提交候选补丁”。本轮仅完成调研和架构重设计，没有修改生产代码或既有开发计划。

## 已确认约束

- [x] AutoDesignMaker 的核心必须面向通用游戏项目，PvZ 只能作为覆盖验证样例，不能提供核心字段、默认值、评分规则或流程分支。
- [x] 正式流程不使用 CrewAI、角色对话、角色互审、AI 投票或由 AI 自主选择下一阶段。
- [x] AI 输出是不可直接写入主状态的候选补丁；唯一的规范存储和编译器负责校验、提交、版本化和追踪。
- [x] 通过确定性规则激活能力模块，以实体/组件、动作、条件/效果、状态、资源、内容、交互、呈现、技术约束和验收场景构成引擎无关 Game IR。
- [x] GitHub 参考项目必须高于 5K stars；低于门槛的项目不得作为方案依据。

## 本地诊断

- 当前 `ArchetypeCatalog` 直接内置 `tower_defense`，并以类型关键词选择需求模板。
- 当前通用 playable 合同仍硬编码 Unity、`DemoScene.unity`、UGUI、固定 HUD 和统一进度状态。
- 当前 D1 项目画像读取的是 AutoDesignMaker 自身应用配置，而不是待生成游戏的产品意图。
- 当前 AI 访谈路由按问题组数量轮换 MDA 阶段，不能证明它是由项目缺口和依赖关系驱动。

## 推荐重构方向

1. 设计阶段改为：项目意图与约束 -> 能力画像 -> 动态模块图 -> 决策图 -> Game IR -> 场景与验收 -> 编译冻结。
2. 现有 16 域/103 节点/515 检查项降级为可检索的知识与问题库，不再默认全部激活或以“存在即完成”计分。
3. Step00-14 保留独立阶段形态，但重写为确定性规范编译、生产 DAG、真实执行和端到端证据链。
4. 使用多个结构差异显著的基准项目做回归；PvZ 只占其中一个样例。
5. 用重复运行、黄金样例、变异测试和真实引擎证据衡量稳定性，不使用 AI 自评或静态字段存在性制造高分。
